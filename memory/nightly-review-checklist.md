# Nightly Review Checklist

每天 24:00 回顾时，优先检查这些会影响第二天连续性的落点。

## 1. 当天新增或验证通过的能力
- 有没有“已经打通/已经验证”的能力还只存在聊天里
- 是否需要补成 skill、workflow 文档、示例、脚本说明
- 若今天修过链路，至少写清楚：做成了什么、如何复用、下次应该先想到什么

## 2. 当天形成的新事实、偏好、结论
- 更新 `memory/YYYY-MM-DD.md`
- 需要长期保存的，再更新 `MEMORY.md`
- 若是持续项目，优先补到专项档案，不要只散落在 daily memory

## 3. 当天推进过的长期项目
- 连载/项目/集成是否已同步正文、索引、状态文件
- 判断进度时，避免只看聊天或单一文件，尽量让档案彼此一致
- 若出现“正文在，索引没跟上”或“skill 建了但记忆没写”的半落盘状态，今晚补齐

## 4. 交付收口
- 写完后检查 git diff，确认关键变更已入库
- 若工作区本来就有无关改动，nightly review 提交时应只暂存本次回顾实际新增/修正的文件，避免把旧实验文件、输出产物或未完成草稿一并卷进提交
- 提交 commit，提交信息应能说明这次夜间回顾沉淀了什么
