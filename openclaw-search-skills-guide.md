# OpenClaw 搜索增强 Skills 清单（可直接给别人看）

这不是“把视频里提到的名字都装一遍”的清单。

这是按**真实可用性、维护风险、配置门槛、以及实际工作流价值**整理出来的一份搜索增强指南。目标很简单：

- 让 OpenClaw 搜得更准
- 让结果来源更多样
- 让它在 API 搜不到时，能进一步进入网页自己找

如果你只想知道结论，先看这一句：

> **推荐路线：先上 Tavily，再补 Multi-Search-Engine，最后再建设浏览器自动化。**

---

## 一、先讲结论：这三类 skill 分别解决什么问题？

### 1. Tavily Search
解决的是：**把默认网页搜索升级成更像“研究型搜索”**。

适合：
- 查公开网页信息
- 找最近新闻或实时内容
- 限定域名搜索
- 抽取页面正文
- 做轻量研究任务

一句话理解：

> **它是“更聪明的搜索 API”。**

---

### 2. Multi-Search-Engine
解决的是：**不要只依赖一个搜索源，而是同时借用多个搜索引擎的覆盖面。**

适合：
- 中文互联网搜索
- 海外资料搜索
- 隐私搜索
- 站内搜索 / filetype / site 限定
- 同一问题多引擎对比

一句话理解：

> **它是搜索调度层，不是单一搜索引擎。**

---

### 3. 浏览器自动化类（browser-use / agent-browser / BrowserMCP 同类）
解决的是：**当搜索 API 或公开页面不够用时，让 agent 直接进网页自己点、自己看、自己找。**

适合：
- 登录后内容
- 动态网页
- 站内搜索
- 表单填写
- 页面截图、页面操作、信息提取

一句话理解：

> **它不是“搜”，而是“亲自进网站处理”。**

---

## 二、推荐优先级

### 第一梯队：建议优先装

#### A. Tavily
**推荐指数：9/10**

为什么先装它：
- 官方生态里就有 Tavily 扩展/skill 线索
- 功能边界很清晰
- 适合立刻提升搜索质量
- 还支持内容抽取，不只是搜结果列表

已确认的信息：
- 官方 OpenClaw 仓库存在 `extensions/tavily/skills/tavily/SKILL.md`
- 官方 skill 描述明确包含：
  - `web_search`
  - `tavily_search`
  - `tavily_extract`
- 支持高级选项：搜索深度、topic、域名过滤、time range、AI answer、内容提取

适合谁：
- 想把 OpenClaw 的外网搜索先拉到“能打”的水平
- 想做新闻、资料、网页研究
- 不想一开始就折腾浏览器自动化

使用前提：
- **需要 Tavily API key**
- 通常还要配 plugin / env

我的评价：
- **这是最应该优先装的一类。**
- 它不是花哨能力，而是基础能力升级。

---

#### B. Multi-Search-Engine
**推荐指数：8/10**

为什么值得装：
- 单一搜索源永远会有盲区
- 中文内容、海外内容、隐私搜索、知识类查询，本来就该分开处理
- 对 OpenClaw 来说，多搜索引擎编排很合理

我查到的主流版本： 
- `multi-search-engine`
- `multi-search-engine-pro`
- 还有其他变体版本

其中比较值得关注的是两类：

##### 版本 1：基础版 multi-search-engine
特点：
- 17 个搜索引擎（8 个中文 + 9 个国际）
- 不需要 API key
- 通过直接构造搜索 URL + `web_fetch` 使用
- 支持 `site:`、`filetype:`、时间过滤、DuckDuckGo bangs、WolframAlpha 查询

优点：
- 上手门槛低
- 不依赖外部付费 API
- 中文/国际混合检索很方便

缺点：
- 本质更像“搜索 URL 模板集合”
- 不是特别智能
- 某些搜索引擎页面结构一变，效果就可能波动

##### 版本 2：multi-search-engine-pro
特点：
- 明显更像“搜索编排工具”而不只是模板列表
- 强调：
  - 区域感知
  - 隐私优先选项
  - 对比模式
  - 安全 URL 构造
  - explain why
- 还带脚本和资源文件，像个完整 toolkit

优点：
- 思路更成熟
- 更接近真正的 research workflow
- 适合认真做搜索策略的人

缺点：
- 结构更重
- 附带文件更多
- 更像“工程工具包”，不一定是最轻的选择

我的建议：
- 如果你想**先上一个稳、直接、轻量的版本**：先试基础版 `multi-search-engine`
- 如果你想做成**长期维护的研究型搜索系统**：再看 `multi-search-engine-pro`

我的评价：
- **它值得装，但不要一口气装多个相似版本。**
- 选一个作为你的“多引擎入口”就够了。

---

### 第二梯队：能力很强，但建议作为第二阶段建设

#### C. browser-use / agent-browser / 类 BrowserMCP skill
**推荐指数：7/10（潜力高，门槛也高）**

为什么强：
- 可以真正打开页面
- 可以点击、输入、截图、提取 HTML / 文本
- 可以处理动态页面和登录态内容
- 当搜索 API 失效时，它往往是最后的兜底能力

我查到的线索：
- OpenClaw skills 归档里有 `browser-use`
- GitHub 上存在多个 browser-use skill 实现
- ClawHub 上存在 `agent-browser` 这类条目

从 skill 描述看，这一类通常支持：
- 打开网页
- 获取页面状态
- 点击元素
- 输入文本
- 截图
- 读取 HTML / text / attributes
- 使用真实 Chrome profile
- 连接已有 Chrome（CDP）

为什么我没有把它排第一：
- 它不是“装了 skill 就自动可用”
- 它依赖本机浏览器、profile、调试能力、自动化环境
- 很容易遇到：
  - profile 被占用
  - 无头模式异常
  - 页面元素不稳定
  - 登录态冲突

但为什么仍然推荐：
- 因为它能解决**搜索类 skill 解决不了的问题**
- 比如：站内搜索、登录后页面、复杂动态页面、实际点按交互

我的评价：
- **这是高上限能力，不是起步必装能力。**
- 适合第二阶段建设。

## 三、到底哪些可以“直接拿来用”？

### 可以直接考虑安装的

#### 1. Tavily
前提：
- 你有 Tavily key
- 你愿意配 env / plugin

结论：
- **可以装**
- **而且很值得装**

#### 2. 基础版 Multi-Search-Engine
前提：
- 你接受它主要是“搜索 URL 构造 + web_fetch 路线”

结论：
- **可以装**
- 不需要 key，试错成本低

---

### 需要配置后才能用好的

#### 3. multi-search-engine-pro
前提：
- 你愿意接受它是 toolkit，而不是一个极简 skill
- 你愿意看资源文件和脚本

结论：
- **能装，但更适合认真折腾的人**

#### 4. browser-use / agent-browser
前提：
- 机器环境要通
- 浏览器依赖要通
- profile / CDP / Playwright / CLI 至少一条链路得通

结论：
- **能装，但不能保证“装完就稳”**

---

### 暂时不建议的做法

#### 不建议 1：同类 skill 一口气装 3~5 个
原因：
- 职责重叠
- 容易抢调用场景
- 出问题时很难判断是谁的问题

#### 不建议 2：把浏览器自动化当成第一层搜索手段
原因：
- 太重
- 太慢
- 太容易被页面变化影响

#### 不建议 3：看视频里谁提得多就装谁
原因：
- 很多视频讲的是“概念正确”
- 但并没有替你筛选维护状态、依赖、兼容性

---

## 四、我推荐的安装路线

### 路线 A：实用优先（最推荐）
适合大多数人。

1. 装 **Tavily**
2. 装 **一个 Multi-Search-Engine**（先基础版）
3. 先用一阵子
4. 真的遇到网页交互瓶颈，再上浏览器自动化

优点：
- 成本低
- 见效快
- 系统不会一下子太复杂

---

### 路线 B：研究型用户
适合愿意认真调搜索策略的人。

1. Tavily
2. Multi-Search-Engine-Pro
3. 浏览器自动化类 skill
4. 再补自己常用站点的专门 skill

优点：
- 上限高
- 研究能力强
- 能形成比较完整的信息获取链

缺点：
- 配置更复杂
- 维护成本更高

---

## 五、一个更现实的工作流示例

如果你真的想让 OpenClaw 把搜索做到极致，推荐让它按下面这条链路工作：

### 第一步：先用 Tavily 做主搜索
适合：
- 查公开资料
- 查新闻
- 限定域名
- 做内容抽取

### 第二步：用 Multi-Search-Engine 做补盲和对比
适合：
- 中文内容补充
- 海外结果对比
- filetype / site 精准搜索
- 低成本扩展覆盖面

### 第三步：用浏览器自动化做“最后一公里”
适合：
- 登录后页面
- 动态网页
- 站内搜索
- 要点进去才能看到的内容

一句话总结就是：

> **API 搜索负责广，Multi-Search 负责补盲，浏览器自动化负责落地。**

---

## 六、给安装者的最后建议

如果你现在刚开始，不要贪多。

### 最稳的选择
只装这两个：
- **Tavily**
- **一个 Multi-Search-Engine**

这已经足够让搜索能力有明显提升。

### 什么时候再上浏览器自动化？
当你开始频繁遇到下面这些情况：
- 搜索结果有，但网页内容不好抓
- 必须登录后才能看
- 要站内搜
- 需要实际点击、输入、翻页

那时再上，不晚。

---

## 七、我的最终建议（懒人版）

如果你只要一句结论：

### 现在就装
- **Tavily**
- **基础版 Multi-Search-Engine**

### 之后再考虑
- **multi-search-engine-pro**
- **browser-use / agent-browser / 类 BrowserMCP**

### 不要做的事
- 不要一次装多个功能重复的搜索 skill
- 不要把浏览器自动化当默认搜索手段
- 不要盲跟视频标题党全装一遍
